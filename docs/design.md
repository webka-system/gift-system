# gift-system 設計書

ソーシャルギフト（ギフトカード / QR 方式）システムの設計ドキュメント。
本ドキュメントは開発の起点となる仕様書であり、Claude Code はこの設計に沿って実装を進める。

---

## 1. システム概要

物理ギフトカードに印刷した QR コードを起点とする、ソーシャルギフトの受け取り・商品選定システム。

購入者がギフトカード（価格帯ごとの商品）を購入すると、あらかじめ用意された QR コード付きの物理カードが発送される。受け取り者はその QR コードを読み取り、価格帯に応じた商品ラインナップから 1 つを選び、配送先住所を入力する。住所が確定した時点で、ネクストエンジン（以下 NE）へ受注データが渡り、実際の商品が発送される。

### このシステムがやらないこと（スコープ境界）

- **MakeShop / 楽天 / Amazon など各モールの API とは連携しない。** 受注取得のポーリングも不要。物理カードの発送は既存の EC 運用に完全に任せる。
- QR カードと元の購入受注をシステム的に紐づけない。突合が必要な場合は管理画面の memo 欄に手入力で対応する。
- メッセージカード作成・メッセージ選択機能は持たない（URL 方式で必要になる機能。本システムには含めない）。

### なぜシンプルか

QR コードを受注と切り離して先に生成する運用のため、住所未確定のまま宙に浮く受注が存在しない。物理カード発送は通常の EC 受注そのものであり、システムが介在しない。NE へ流すのは受け取り者が住所を確定したときだけで、その時点で住所は必ず揃っている。この結果、モール API 連携・受注ポーリング・期限切れフォールバックといった複雑な要素がすべて不要になっている。

## 2. 業務フロー（確定版）

```
[管理側]
カード種別登録（例：1万円 / 2万円 / 3万円）
  └ 各種別に選定可能商品を登録（画像・商品説明・NE用商品コード）
QRコードを種別ごとに一括生成（例：3万円カードを100個）
  └ 印刷用PDF/画像を出力 → 物理ギフトカードへ印刷
注文が来たら在庫カードを都度発送（既存EC運用、システム介在なし）
QR一覧でmemo記入（受注番号など任意・突合用）

[受け取り者側]
QRコード読み取り → トークン付きURLで受け取り者画面に着地
  └ カード種別に応じた商品ラインナップを表示（画像＋説明）
商品を1つ選択 → 配送先住所を入力 → 確定
  └ Cloud FunctionがNEへ受注登録（自動）またはCSV出力（手動）
  └ 該当QRカードを「使用済」に変更（二重利用防止）

[NE / ネクストエンジン]
住所付き受注を受け取り、在庫引き当て・発送
```

### フォールバック・期限管理について

URL 方式で必要だった「一定期間住所未確定なら購入者住所へ発送」というフォールバックは、**本 QR 方式では不要**。物理カードは既に購入者住所へ届いており、宙に浮く受注が存在しないため。

## 3. データモデル（Cloud Firestore）

3 つのコレクションで構成する。中心にあるのは「価格帯（親）とその中の選択肢（子）」という入れ子構造。

### 3.1 giftCardTypes（ギフトカード種別 / 親）

価格帯ごとに 1 レコード。

| フィールド | 型 | 説明 |
|---|---|---|
| id | string | ドキュメントID |
| name | string | 表示名（例：「3万円ギフトカード」） |
| price | number | 価格帯（例：30000） |
| cardProductCode | string | ギフトカード側の管理商品コード |
| createdAt | timestamp | 作成日時 |
| active | boolean | 有効/無効 |

### 3.2 selectableProducts（選定可能商品 / 子）

各カード種別にぶら下がる、受け取り者が選べる商品。**同じ商品が複数の価格帯にまたがることはない**ため、種別配下にぶら下げる素直な構造とする（種別への単純な参照でよく、多対多は不要）。

| フィールド | 型 | 説明 |
|---|---|---|
| id | string | ドキュメントID |
| cardTypeId | string | 所属するgiftCardTypesのID |
| name | string | 商品名 |
| description | string | 簡単な商品説明 |
| imageUrl | string | 商品画像URL（Firebase Storage） |
| neProductCode | string | 選定されたときNEへ流す実商品コード |
| active | boolean | 有効/無効 |

### 3.3 giftCards（発行済みQRカード）

先に一括生成しておくカード。生成時点では「まだ誰のものでもない空のカード」。受け取り者が使用して初めて受注として立ち上がる。

| フィールド | 型 | 説明 |
|---|---|---|
| id | string | ドキュメントID |
| token | string | 推測不可能なユニークトークン（URL用） |
| cardTypeId | string | どの価格帯のカードか |
| status | string | `unused`（未使用） / `used`（使用済） |
| memo | string | 管理者が手入力する自由記入欄（受注番号など突合用） |
| createdAt | timestamp | 生成日時 |
| ── 以下は使用時に書き込まれる ── | | |
| selectedProductId | string | 受け取り者が選んだ商品 |
| shippingAddress | object | 配送先住所（氏名・郵便番号・住所・電話など） |
| usedAt | timestamp | 使用（確定）日時 |
| neStatus | string | NE投入状態（未投入 / 投入済 / CSV出力済 など） |

**1 カードで選べる商品は 1 つのみ**（複数組み合わせは不可）。

## 4. 機能一覧

### 4.1 管理画面（要ログイン）

- **カード種別管理**：価格帯（種別）の登録・編集・有効/無効切り替え
- **選定可能商品管理**：各種別への商品登録・編集（画像アップロード、商品説明、NE用商品コード）
- **QRコード一括生成**：種別を指定して QR を任意個数生成（例：3万円カードを100個）
- **印刷用出力**：生成した QR を印刷用 PDF / 画像として出力
- **QR一覧管理**：ステータス（未使用/使用済）確認、memo 入力
- **受注確認**：どのカードでいつ何が選ばれ、どこへ送られたか、NE 投入状態の確認

### 4.2 受け取り者画面（ログイン不要・トークンURLで着地）

- QR 読み取りでトークン URL に着地
- カード種別に応じた商品ラインナップ表示（画像＋説明）
- 商品を 1 つ選択
- 配送先住所入力
- 確定 → 完了画面
- 使用済みトークンで再アクセスした場合は「使用済み」表示（二重利用防止）

## 5. 技術スタック

シールデザインシステムと同一構成に揃える。

| 領域 | 採用技術 |
|---|---|
| クラウド基盤 | GCP / Firebase |
| データベース | Cloud Firestore（ロケーション：asia-northeast1 / 東京） |
| ホスティング | Firebase Hosting（管理画面・受け取り者画面） |
| 画像保存 | Firebase Storage（商品画像） |
| サーバー処理 | Cloud Functions |
| 認証 | Firebase Authentication（管理画面のみ。受け取り者側はトークンURLで不要） |
| 課金プラン | Blaze（Cloud Functions からの外部通信＝NE API 呼び出しに必須） |
| ソース管理 | GitHub（`gift-system` リポジトリ） |

> フロントエンドのフレームワークやビルドツールの詳細は、シールデザインシステムのリポジトリ **`label-system`** の構成に準拠する。実装開始時に `label-system` を参照してスタックを揃えること。

## 6. ネクストエンジン（NE）連携

**自動・CSV の両対応**とする。片方に決め打ちせず、両方の口を用意しておく。

- **自動連携**：受け取り者の住所確定をトリガーに、Cloud Function から NE の受注登録 API を呼び出して住所付き受注を投入する。NE API のトークンはリフレッシュ管理が必要な点に注意。
- **CSV 連携**：確定済み受注を CSV として出力し、手動または日次で NE へ取り込む。

いずれの方式でも、NE へ投入するのは **住所が確定した実商品受注のみ**。投入後は giftCards の neStatus を更新する。

## 7. メール通知方針

**受け取り者へのメール（注文受付・発送通知）は NE のメール機能に任せる。本システムでは自前のメール送信（SendGrid 等）を実装しない。**

- 注文受付・発送完了などのメールは、住所確定後に NE へ受注が入った時点で NE 側の自動メール機能から送られる。したがって SendGrid によるリアルタイム送信の仕組みは不要。
- label-system（シール）が SendGrid を使っているのは、シール側に NE のようなメール代行基盤がなくデザイン確定通知等を自前で送る必要があるため。本システムは NE がその役割を担うため事情が異なる。
- **即時の安心感は「完了画面」で担保する。** 受け取り者が住所入力を終えた瞬間に、画面上で「ご注文を受け付けました。確認メールは追ってお送りします」と明確に表示する。メール＝事後確認、画面＝即時の安心、と役割を分ける。
- 将来、即時受付メールが必要になった場合に SendGrid 等を後から追加することは可能。ただしその際は NE の受付メールとの二重送信に注意する。当面は追加しない。

## 8. セキュリティ・運用上の注意

- **サービスアカウントの鍵 JSON・API キー・.env は絶対にコミットしない。** `.gitignore` に必ず含める。
- Firestore はテストモードではなく本番モードで運用し、セキュリティルールを明示的に記述する。
- 受け取り者トークンは推測不可能なランダム値とする（外部アクセス制御の要）。
- GCP コンソールで予算アラートを設定し、想定外の課金を検知できるようにする。
- Cloud Firestore のロケーションは変更不可のため東京（asia-northeast1）で確定。

## 9. 開発の進め方

1. 本設計書を `docs/design.md` としてリポジトリに配置（第一コミット）
2. `firebase init` でリポジトリと gift-system プロジェクトを紐づけ（Firestore / Hosting / Functions を選択）
3. データモデル（3 コレクション）と Firestore セキュリティルールを実装
4. 管理画面（カード種別・商品・QR 生成・一覧）を実装
5. 受け取り者画面（トークン着地・商品選択・住所入力）を実装
6. NE 連携（自動 API / CSV 出力の両対応）を実装
7. QR 生成・印刷用 PDF 出力を実装

開発の各段階で進捗と決定事項を記録し、後から仕様書としてまとめられる状態を保つ。
